## 二、此前报告已记录、经复核仍存在问题（或表现形式已变化）

### 1. ModDamageTypes — 伤害类型注册完全缺失，运行时将导致崩溃

- **严重程度**: **Critical**
- **状态**: 此前报告 #1，表现形式已变化（`DeferredRegister` 已被完全移除，但核心问题恶化）
- **文件**: `src/main/java/com/samplecat/atlantisorigins/common/damage/ModDamageTypes.java`
- **行号**: 第 18–40 行
- **相关文件**: `src/main/java/com/samplecat/atlantisorigins/common/event/DcsEventHandler.java`（第 137、161 行）、`PressureEventHandler.java`（第 151、153 行）、`HypothermiaEventHandler.java`（第 136 行）
- **问题描述**:
  1. 当前代码中 `ModDamageTypes` 仅包含 `ResourceKey<DamageType>`（`TRUE_DAMAGE`、`DCS_DAMAGE`、`EVOLUTION`）和一个 `bootstrap(BootstrapContext)` 方法，**没有任何 `DeferredRegister`**，也**没有任何地方调用 `bootstrap`**。
  2. 经文件系统检查，`data/atlantis_origins/damage_type/` 目录下**没有任何 JSON 文件**。
  3. 在 Minecraft 1.21.1 中，`DamageType` 必须通过 JSON 数据包或 `DatapackBuiltinEntriesProvider` 注册。当前两种机制均未实现，导致 `Registries.DAMAGE_TYPE` 中不存在这三个条目。
  4. `DcsEventHandler`、`PressureEventHandler`、`HypothermiaEventHandler` 均在运行时调用 `entity.damageSources().source(ModDamageTypes.XXX)`。在 Minecraft 中，`DamageSources.source(ResourceKey)` 内部执行 `this.damageTypes.getOrThrow(key)`。如果 `DamageType` 未注册，`getOrThrow` 会抛出 **`NoSuchElementException`**（`Missing key ...`），导致**服务器直接崩溃**。
  5. `ModDamageSources.java` 第 12 行也使用了 `ModDamageTypes.EVOLUTION`，同样会触发崩溃。
  6. 此前报告描述为"`DeferredRegister` 为空"，但当前代码中 `DeferredRegister` 已完全被移除，问题从"注册器空转"恶化为"完全没有注册机制"。
- **修复建议**:
  - **紧急方案**：在 `src/main/resources/data/atlantis_origins/damage_type/` 下手动创建 `true_damage.json`、`dcs_damage.json`、`evolution.json` 三个 JSON 文件。
  - **长期方案**：在数据生成中通过 `DatapackBuiltinEntriesProvider` 调用 `ModDamageTypes.bootstrap()`，确保 JSON 在构建时自动生成。
  - **兜底方案**：在调用 `damageSources().source()` 前添加 `if (entity.level().registryAccess().lookupOrThrow(Registries.DAMAGE_TYPE).containsKey(key))` 检查，防止崩溃。

### 2. PoseidonsAnvilHandler — 此前报告 #5 描述不准确，但存在数据清理副作用

- **严重程度**: **Medium**
- **状态**: 此前报告 #5 描述"没有任何代码设置 `unfinished` 标记"**不准确**，但存在新的数据清理问题
- **文件**: `src/main/java/com/samplecat/atlantisorigins/common/event/PoseidonsAnvilHandler.java`
- **行号**: 第 28 行、第 34–40 行
- **相关文件**: `src/main/java/com/samplecat/atlantisorigins/common/menu/CupriteWeaponStationMenu.java`（第 104–106 行）
- **问题描述**:
  1. 经复核，`CupriteWeaponStationMenu.java` 第 104–106 行确实会设置 `"unfinished": true`：
     ```java
     CompoundTag tag = new CompoundTag();
     tag.putBoolean("unfinished", true);
     result.set(DataComponents.CUSTOM_DATA, CustomData.of(tag));
     ```
     因此 `PoseidonsAnvilHandler` 的 `"unfinished"` 检查**不是死代码**，此前报告描述有误。
  2. **新问题**：第 28 行 `output.remove(DataComponents.CUSTOM_DATA)` 会**移除整个 `CUSTOM_DATA` 组件**。如果物品通过其他机制（如第三方模组、数据包、命令）附加了额外的自定义数据，这些数据也会被一并清除。虽然当前代码中 `CUSTOM_DATA` 只包含 `"unfinished"` 标记，但这种"一刀切"的移除方式存在安全隐患。
- **修复建议**: 将 `output.remove(DataComponents.CUSTOM_DATA)` 改为仅移除 `"unfinished"` 标记（例如通过 `CustomData.update(...)` 或复制 tag 后删除特定键），而不是移除整个 `CUSTOM_DATA` 组件。

## 三、新发现的问题

### 3. ModEntities — 方法引用类型不匹配导致生成逻辑隐患

- **严重程度**: **Medium**
- **状态**: 新发现（此前报告 #12 的显式转换已移除，但方法引用仍存在相同逻辑问题）
- **文件**: `src/main/java/com/samplecat/atlantisorigins/common/entity/ModEntities.java`
- **行号**: 第 95–119 行
- **问题描述**:
  1. `AbyssalCrab`（深海蟹）使用 `Axolotl::checkAxolotlSpawnRules` 作为生成检查（第 104 行）。
  2. `AbyssalJellyfish`、`StingingJellyfish`、`GazingJellyfish` 使用 `Squid::checkSurfaceWaterAnimalSpawnRules`（第 97、110、112 行）。
  3. `BioluminescentFish`、`DeepEel`、`Gulper`、`KrillSwarm` 使用 `AbstractFish::checkSurfaceWaterAnimalSpawnRules`（第 100、102、106、108 行）。
  4. 虽然 Java 类型擦除允许这些编译通过，但这些静态方法的**设计意图**是为特定实体类型（`Axolotl`、`Squid`、`AbstractFish`）服务的。如果未来 Minecraft 版本在这些方法中增加了实体类型特定的检查（例如通过 `type` 参数反射获取实体类、或调用 `type.create()` 来验证），当前代码会**立即崩溃**或产生不可预期的行为。
  5. 例如，如果 `Axolotl.checkAxolotlSpawnRules` 内部某天添加了 `((Axolotl)type.create(...))` 或 `type.getBaseClass() == Axolotl.class` 的检查，`AbyssalCrab` 的生成将直接失败或抛出 `ClassCastException`。
- **修复建议**: 为每个自定义实体编写独立的生成检查逻辑，或使用 `SpawnPlacements` 的标准位置/环境检查（如 `SpawnPlacementTypes.IN_WATER` 已处理水环境检查），避免依赖其他实体类型的静态方法作为方法引用。

## 四、问题总结与优先级建议

| 优先级 | 问题 | 严重程度 | 说明 |
|--------|------|---------|------|
| 🔴 **最高** | ModDamageTypes 缺少注册机制 | **Critical** | 没有 JSON 文件，`bootstrap` 未调用，`damageSources().source()` 会在运行时抛出 `NoSuchElementException`，导致服务器崩溃。涉及 `DcsEventHandler`、`PressureEventHandler`、`HypothermiaEventHandler` 和 `ModDamageSources`。 |
| 🟠 **高** | PoseidonsAnvilHandler 数据清理副作用 | **Medium** | `remove(DataComponents.CUSTOM_DATA)` 会清除所有自定义数据，存在误删风险。 |
| 🟠 **高** | ModEntities 方法引用类型不匹配 | **Medium** | 使用 `Axolotl::`、`Squid::`、`AbstractFish::` 的静态方法作为其他实体类型的生成检查，未来版本兼容性风险。 |
| 🟡 **低** | ModDamageTypes 文档不一致 | **Low** | `content_registry.md` 遗漏 `EVOLUTION` 伤害类型。 |

### 修复顺序建议

1. **立即处理 Critical 问题**：为 `true_damage`、`dcs_damage`、`evolution` 创建 `data/atlantis_origins/damage_type/*.json` 文件，或在数据生成中调用 `ModDamageTypes.bootstrap()`。这是当前唯一会导致**服务器崩溃**的问题。
2. **修复 PoseidonsAnvilHandler**：将 `output.remove(DataComponents.CUSTOM_DATA)` 改为仅移除 `"unfinished"` 键，避免误删其他自定义数据。
3. **重构 ModEntities 生成检查**：将 `Axolotl::`、`Squid::`、`AbstractFish::` 的方法引用替换为自定义的 `SpawnPredicate` 实现，消除类型兼容性隐患。
4. **同步文档**：在 `content_registry.md` 中补充 `evolution` 伤害类型条目。

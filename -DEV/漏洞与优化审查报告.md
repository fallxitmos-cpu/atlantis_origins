# Atlantis Origins 项目漏洞与优化审查报告

> 审查日期：2026-06-21  
> 审查版本：`0.2.5-alpha-fix6`  
> 审查范围：Java源代码、资源文件、数据包、配置系统  
> 原则：仅指出问题，不修改代码

## 🔴 严重漏洞（运行时风险 / 逻辑错误）

### 1. ModDamageTypes — 注册器空转 + bootstrap 未调用

**文件**：`common/damage/ModDamageTypes.java`（第 9–35 行）  
**问题**：`DeferredRegister<DamageType> DAMAGE_TYPES` 被创建了，但 `TRUE_DAMAGE`、`DCS_DAMAGE`、`EVOLUTION` 都是 `ResourceKey` 而不是 `DeferredHolder`，从未被 `register` 到 `DAMAGE_TYPES` 中。`AtlantisOrigins` 第 57 行调用了 `ModDamageTypes.DAMAGE_TYPES.register(modEventBus)`，但这注册了一个**空注册器**。

`bootstrap(BootstrapContext<DamageType>)` 方法用于 `DatapackBuiltinEntriesProvider` 数据生成，但项目中**没有任何地方调用它**。这意味着：
- 如果 `data/atlantis_origins/damage_type/` 下的 JSON 文件被误删或损坏，伤害类型将完全失效
- 数据生成时不会自动产出这些 JSON
- `DeferredRegister` 和 `bootstrap` 两套机制混用，彼此不衔接

**修复方向**：
- 方案 A：在 `AtlantisOrigins` 中通过 `DatapackBuiltinEntriesProvider` 调用 `ModDamageTypes.bootstrap`
- 方案 B：移除 `DeferredRegister`，仅保留 `ResourceKey` 和 `bootstrap`，并确保数据生成覆盖
- 方案 C：使用 `DAMAGE_TYPES.register(...)` 把 `ResourceKey` 真正注册进去（NeoForge 1.21.1 对 `DamageType` 的 `DeferredRegister` 支持有限，需验证）

### 2. OrichalcumArmorEventHandler — 配置项被忽略

**文件**：`common/event/OrichalcumArmorEventHandler.java`（第 18–41 行）  
**问题**：`Config.ORICHALCUM_ARMOR_EFFECTS_ENABLED` 在 `Config.java` 中已定义，但 `OrichalcumArmorEventHandler` 中**从未读取该配置**。无论玩家是否关闭配置，山铜盔甲的减伤效果始终生效。

**修复方向**：在 `onLivingDamagePre` 开头添加 `if (!Config.ORICHALCUM_ARMOR_EFFECTS_ENABLED.get()) return;`

### 3. ModNetwork — ToggleRiptidePacket 缺乏安全校验

**文件**：`common/network/ModNetwork.java`（第 70–92 行）  
**问题**：`handleToggleRiptide` 在收到客户端发来的 `ToggleRiptidePacket` 后，直接遍历玩家双手的物品，如果持有 `ORICHALCUM_TRIDENT` 就切换忠诚/激流附魔。但：**服务端没有验证发送者是否确实持有该物品**。如果客户端被修改，可以伪造这个包来任意切换附魔。

**修复方向**：在服务端 handler 中，添加 `if (!player.getItemInHand(hand).is(ModItems.ORICHALCUM_TRIDENT.get())) return;` 的双重校验（虽然已有遍历，但应确保遍历到的物品确实匹配）。

### 4. TridentEventHandler — 80%暴击率 + 无侧校验

**文件**：`common/event/TridentEventHandler.java`（第 13–47 行）  
**问题**：
- `CRIT_CHANCE = 0.80F`（80%概率），`CRIT_MULTIPLIER = 1.5F`。投掷山铜三叉戟有 80% 概率造成 22.5 伤害（15 × 1.5），这在游戏平衡上极其夸张。
- 没有检查 `event.getEntity().level().isClientSide()`，虽然 `LivingIncomingDamageEvent` 通常只在服务端触发，但显式校验更安全。

**修复方向**：
- 将 `CRIT_CHANCE` 降至合理值（如 0.15–0.25）
- 添加 `if (event.getEntity().level().isClientSide()) return;`

### 6. ModAttachments — 大多数附件未持久化 / 未跨死亡保留

**文件**：`common/attachment/ModAttachments.java`（第 14–56 行）  
**问题**：
- `EVOLVED` 有 `.serialize(Codec.BOOL).copyOnDeath().sync(ByteBufCodecs.BOOL)` — 这是正确的
- 但 `PLAYER_OXYGEN`、`HYPOTHERMIA_TICKS`、`PRESSURE_STAGE`、`DCS_STAGE`、`DCS_TIMER` 等**均没有 `copyOnDeath()`**。这意味着玩家死亡重生后，氧气量、失温累计、DCS 状态等全部丢失，回到默认值。
- 对于氧气系统，重生后氧气回到 300 满值可能不是大问题；但 DCS 状态丢失可能导致玩家重生后莫名其妙地治愈了减压症。

**修复方向**：为所有需要在死亡后保留的附件添加 `.copyOnDeath()`，并根据设计意图决定哪些应该保留、哪些应该重置。

### 8. PotionEventHandler — 只拦截原版药水，漏过其他来源

**文件**：`common/event/PotionEventHandler.java`（第 17–54 行）  
**问题**：`onUseItemStart` 只检查 `stack.getItem() == Items.POTION`，即原版水瓶药水。但：
- 喷溅型水肺药水（`SPLASH_POTION`）
- 滞留型水肺药水（`LINGERING_POTION`）
- 其他模组通过 `ItemStack` 给予的水肺效果
- 直接通过 `/effect` 命令或药水云施加的水肺效果

这些都无法被拦截。`OxygenEventHandler` 第 47 行会在玩家装备氧气罐时移除 `MobEffects.WATER_BREATHING`，但 `PotionEventHandler` 只拦截了喝药水的动作，没有拦截所有来源。

**修复方向**：在 `OxygenEventHandler` 的 `onPlayerTick` 中，改为每 tick 检查并移除 `WATER_BREATHING`（当前只在喝药水时移除），或者扩展 `PotionEventHandler` 监听 `MobEffectEvent.Added`。

### 9. OxygenEventHandler — 线程不安全的 Random 实例

**文件**：`common/event/OxygenEventHandler.java`（第 22–23 行）  
**问题**：`private static final Random RANDOM = new Random();` 是一个全局单例。在多人服务器环境中，多个玩家的 `PlayerTickEvent` 可能在不同线程上并发触发（虽然 Minecraft 服务端 tick 是单线程的，但 `Random` 的并发访问仍不是好实践）。

**修复方向**：使用 `ThreadLocalRandom.current()` 或 `player.getRandom()`。

### 10. AlchemicalReactorBlockEntity — 配方输入槽范围错误

**文件**：`common/blockentity/AlchemicalReactorBlockEntity.java`（第 147–151 行）  
**问题**：`SimpleItemHandlerRecipeInput` 的 `size()` 返回 `handler.getSlots()`，即 5 个槽位（包括输出槽）。但 `AlchemicalReactorRecipe` 的 `matches` 逻辑可能会检查所有 5 个槽，如果输出槽已有物品，可能导致配方匹配失败。

**修复方向**：将 `size()` 改为 `4`（只包含输入槽），或者确保 `RecipeInput` 的实现正确隔离输入和输出。

### 24. ModDamageTypes — 两套注册机制混用

### 25. AbyssalJellyfish — MethodHandle 的异常风险

**文件**：`common/entity/AbyssalJellyfish.java`（第 35–42 行）  
**问题**：使用 `MethodHandle.unreflectSpecial` 绕过 `Squid#hurt()` 的喷墨行为。虽然这是一个已知的技术方案，但如果未来 Minecraft 版本改变了 `LivingEntity#hurt` 的签名，这个 `static` 初始化块会在类加载时直接抛出 `RuntimeException`，导致整个实体注册失败。

**修复方向**：在 `hurt()` 方法中直接覆盖行为，而不是使用反射。例如：
```java
@Override
public boolean hurt(DamageSource source, float amount) {
    this.setInkSquirtTick(0); // 取消喷墨计时器
    return super.hurt(source, amount);
}
```
如果 `Squid` 的喷墨是在 `hurt()` 中触发的，可以直接控制 `inkSquirtTick` 字段（如果可访问）或复制 `LivingEntity.hurt()` 的调用逻辑。

## 🟢 优化建议（性能 / 设计）

### 27. 事件处理器合并

**问题**：`PressureEventHandler`、`DcsEventHandler`、`HypothermiaEventHandler` 都监听 `EntityTickEvent.Post`，且都检查 `entity instanceof Player`、`!isClientSide`、`!isSpectator`、`!isCreative`。这导致了**重复的条件检查**。

**优化方向**：创建一个统一的 `PlayerTickEventHandler`，在一个 `PlayerTickEvent.Post` 中依次调用 `pressureTick`、`dcsTick`、`hypothermiaTick`，减少事件分发开销。

---

### 28. ModAttachments — 同步策略优化

**问题**：`PLAYER_OXYGEN` 每 10 tick 同步一次（`OxygenEventHandler` 第 90 行），`PRESSURE_STAGE` 每 5 tick 或阶段变化时同步（`PressureEventHandler` 第 74 行）。但 `EFFECT_INTENSITY` 作为 `Float` 每次同步都会发送，即使变化很小。

**优化方向**：只有当 `oxygen`、`pressureStage`、`effectIntensity` 的值**实际变化**时才发送网络包。当前 `PressureEventHandler` 已经在 `newStage != currentStage` 时发送，但 `intensity` 每 5 tick 都会发送。

---

### 29. 配置热重载支持

**问题**：`Config.java` 和 `ClientConfig.java` 中的配置值在运行时通过 `ConfigSpec` 定义，但 `OxygenEventHandler`、`PressureEventHandler` 等事件处理器在每次 tick 都读取 `Config.XXX.get()`，这会导致频繁的 `AtomicReference` 解引用。

**优化方向**：虽然 `ConfigSpec.get()` 性能足够好，但如果某些配置（如 `PRESSURE_ENABLED`）在运行时不应改变，可以在 `FMLCommonSetupEvent` 中缓存到局部变量。

---

### 30. 实体渲染器注册统一

**文件**：`client/ClientSetup.java`（第 86–99 行）  
**问题**：`event.registerEntityRenderer(...)` 使用了完整包名 `com.samplecat.atlantisorigins.common.entity.ModEntities.XXX.get()`，这不如 `import` 后使用短名称清晰。

**优化方向**：添加 `import com.samplecat.atlantisorigins.common.entity.ModEntities;`，然后使用 `ModEntities.ABYSSAL_JELLYFISH.get()`。

---

### 31. 语言文件统一生成

**问题**：`zh_cn.json`、`en_us.json` 等语言文件中有大量手动维护的键。随着物品/方块增加，维护负担会越来越大。

**优化方向**：考虑使用 `data generator` 自动生成语言文件（NeoForge 的 `LanguageProvider`），避免手动同步。
